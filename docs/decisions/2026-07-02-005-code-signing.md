# 005. Code signing = stable self-signed, non-sandboxed

**Date:** 2026-07-02
**Status:** Proposed (applies from Phase 3)
**Deciders:** Claude (CTO, recommends) + User (CEO, approves)

## Context
The app needs Accessibility (AX text injection) + Input-Monitoring (global hotkey), which **require a non-sandboxed app** — so the App Sandbox / `network.client` entitlement cannot be relied on to block network egress (egress is instead enforced by the build-phase binary scan, ADR-004 / privacy-guardian). Original plan / open-question Q5 assumed **ad-hoc** signing for Phases 1–4.

## Problem with ad-hoc
An ad-hoc identity is unstable: its designated requirement changes across rebuilds, so macOS **TCC grants (Accessibility, Input-Monitoring) churn** — the user must re-grant on every rebuild, breeding habituation to re-granting powerful permissions. Any local bundle-writer could also inherit the AX grant.

## Decision
- **A stable self-signed identity** (stable designated requirement) for the **app from Phase 3 onward** — TCC grants persist across rebuilds.
- **Non-sandboxed**, hardened runtime where compatible; egress controlled by the build scan, not entitlements.
- **Ad-hoc** signing is fine only for the throwaway **Phase 1–2 CLI**.
- No paid Apple Developer account needed for personal use; revisit only if sharing the app (Developer-ID + notarization) becomes a goal.

## Consequences
- Create a stable self-signed certificate once; document its use in the Phase 3 spec.
- Reconciles the plan's earlier "ad-hoc" note (`CLAUDE.md` Q5) — updated there when Phase 3 opens.
